\section{Introduction}

\subsection{Purpose of the Language}
Trisynth exists to bridge the gap between high-level expressive programming and ultra-low-level bare-metal execution. Functioning as an educational but fully realized standalone compiler, Trisynth solves the problem of ``black-box'' compilation by handling all custom intermediate representations (IR), aggressive semantic analysis, and direct instruction emission explicitly without external monoliths like LLVM.

\subsection{Design Goals}
\begin{itemize}
    \item \textbf{Transparency}: Every phase of the compiler (Tokens $\rightarrow$ AST $\rightarrow$ IR $\rightarrow$ ASM) can be inspected via CLI flags.
    \item \textbf{Performance}: Capable of Ahead-Of-Time (AOT) compilation utilizing significant optimization techniques (CSE, Constant Folding, Dead-Code Elimination).
    \item \textbf{Safety}: Fully static type resolution, block-scoping, and immutable variables ensure logical stability at compile-time natively.
    \item \textbf{Portability}: Native code generators seamlessly switch outputs between `x86-64` (Intel/AMD) and `RISC-V` architectures natively mapping limits intelligently.
\end{itemize}

\subsection{Key Features and Positioning}
Compared to C, Trisynth offers a strictly predictable runtime subset limiting undefined behaviors linked to raw pointers, preferring statically sized arrays logically locked to explicitly evaluated stacks properly mapping boundaries uniquely allocating structures successfully avoiding leaks natively. Compared to LLVM, it forces students to analyze direct internal AST translations inherently studying components confidently.
